# 北北基垃圾車平台：下一階段分工與前後端接口規格

## 0. 文件目的

本文件用於下一階段開發分工，承接 [ARCHITECTURE.md](ARCHITECTURE.md)。目標：

1. 把工作切成五等份（三人後端、兩人前端），讓五人能同時作業。
2. 盡量避免多人同時修改同一個檔案（降低 Git 衝突）。
3. 明確定義前後端 API 對接規格，避免前端自己猜欄位、後端自己改回傳格式。

> 後端「地基」已建好並通過語法檢查（共用回應 `ok/err`、`auth` 裝飾器、8 個 Blueprint 全註冊、所有端點骨架含 `# TODO(Px)`、LIFF 通用路由）。各組員只需填自己負責檔案內的 `# TODO`。
> 若實作時發現程式碼與本文件不一致，**以本文件的「目標接口規格」為對齊方向**。

## 1. 下一階段總目標

不是加大量新功能，而是把兩條 MVP 閉環跑通：

```text
【一般使用者 / LINE】
加入官方帳號 → 輸入「綁定帳號」→ LIFF 一鍵綁定（存 line_user_id）
→ 查附近站點 / 收運日程 → 收藏定點、設定到站提醒 → 到站前收到 LINE 推播

【管理者 / React 後台】
帳密登入 → 檢視 / 停權使用者、調整權限
→ 發布環保政策公告並推播給目標縣市 / 全體 → 檢視 ETL 同步狀態
```

完成後：LINE 使用者能查站點並收到提醒；管理員能管理使用者、發公告、看同步健康度。

## 2. 核心約定

### 2.1 身分驗證（**本階段＝簡易版**；安全版見第 13 節「以後做」）

| 對象 | 簡易版（現在做） | 安全版（以後做） |
| :-- | :-- | :-- |
| 一般使用者 | 每次請求帶 header `X-Line-User-Id: <LINE userId>`，後端查 `users` | `liff.getIDToken()` → `Authorization: Bearer <idToken>`，後端呼 LINE verify 取 `sub` |
| 管理者 | `POST /api/admin/login` 用 **email + password** 查 DB；之後請求帶 `X-Admin-User: <email>` | 登入發 JWT，請求帶 `Authorization: Bearer <jwt>` |

> ⚠️ 簡易版 header 可偽造，**僅供開發 / Demo**。不能一半簡易、一半安全；要升級就一次性換掉所有驗證。

後端已提供裝飾器（`app/utils/auth.py`），端點直接套用即可：

```python
from app.utils.auth import line_required, admin_required
from flask import g

@bp.route('/', methods=['GET'])
@line_required                      # 驗 X-Line-User-Id；通過後 g.current_user 可用
def list_favorites():
    user_id = g.current_user['user_id']
    ...
```

### 2.2 通用 API 規則

- **Base URL**：`/api`（開發 `http://localhost:5000/api`；LIFF 實測走 ngrok HTTPS）。
- **Content-Type**：`application/json`，UTF-8。
- **回應格式**（後端已用 `app/utils/responses.py` 的 `ok()` / `err()` 統一）：

成功：
```json
{ "status": "success", "data": { } }
```
列表（可帶 `count`）：
```json
{ "status": "success", "count": 2, "data": [ ] }
```
失敗：
```json
{ "status": "error", "message": "錯誤訊息" }
```

- **HTTP 狀態碼**：200 成功、201 新增、400 參數錯、401 未驗證、403 停權/無權、404 找不到、409 衝突、500 伺服器錯、501 尚未實作。
- **前端讀錯誤訊息**：`res.json()` 後取 `body.message`。

```js
const body = await res.json();
if (body.status !== 'success') alert(body.message);
```

## 5. 避免衝突的協作規則

1. 每個檔案只有一個主要負責人；要改別人的檔，先在群組說原因。
2. `app/__init__.py`、`db.py`、`utils/responses.py`、`utils/auth.py` 為共用地基，原則上不動；真要改先講，由一人統一合併。
3. API 回傳欄位若要改，**先更新本文件**（或開接口變更紀錄），再改程式。
4. 前端只「使用」API，不自行假設欄位；以本文件第 7 節為準。
5. 後端不為了配合某個前端頁面臨時改回傳格式；要改走規則 3。

## 6. 任務拆分（Task Breakdown）

### P1 後端·站點與地理

- `geo_service.find_nearby_stations(lat, lng, radius_km, limit)`：先用經緯度 `BETWEEN`（bbox）預過濾，再以 Haversine 計算 `distance_km` 並升序（站點約 17.5 萬筆，**禁止全表 Haversine**）。
- `geo_service.get_station_detail(station_id)`：JOIN routes / areas / station_schedules，組出明細 + 7 天日程。
- `geo_service.next_arrival(station_id)`：依「今天星期幾」往後找最近一筆有收運的日程（`day_of_week` 0=日，Python `weekday()` 一=0，需換算），回傳 `in_minutes`。
- 接上 `routes.py` 四個端點（見 7.2）。
- **完成標準**：`curl "/api/stations/search?lat=25.04&lng=121.51"` 回傳依距離排序的站點。

### P2 後端·使用者資料

- 收藏 CRUD（限本人 `g.current_user['user_id']`；新增重複→409）。
- 通知 CRUD（同上）。
- 資訊三個公開查詢（依 `city` 篩選；公告含 `target_city IS NULL` 或符合者，新到舊）。
- `users.py`：綁定時 `username` 帶入 LINE 暱稱；新增 `PUT /api/users/credentials`（`line_required`，設 email+密碼，email 重複→409）。
- **完成標準**：帶 `X-Line-User-Id` 可完成收藏 / 通知增查改刪，並能自助設定 email+密碼。

### P3 後端·管理與背景

- `line_service`：`reply_text` / `push_text` / `multicast_text`（line-bot-sdk v3），`webhooks.py` 改呼叫。
- `admin.py`：登入（**email + 密碼** `check_password_hash`）、使用者列表 / 停權 / 升降權（**升 admin/developer 前該 user 須已有 email+密碼**）、公告 CRUD + 推播、同步監控。`X-Admin-User` 帶 email。
- `webhooks.py`：偵測傳訊者 `role` 為 admin/developer → 回覆 React 後台網址（`ADMIN_WEB_URL`，需加進 config）；非管理者不給。
- `notifier.py`：每 60 秒掃描，到站前 `remind_before_mins` 內且當日未通知過 → `push_text`（當日去重用記憶體 set，標註多程序限制）。
- `data_sync.py`：每日 02:00 跑 ETL，結束寫 `api_sync_log`。
- 首位管理者：**不需 `create_admin.py`**；人工在 DB 把已設好 email+雜湊密碼的使用者 `role` 改 `admin`（或該人先用 7.1.3 自助設定再改）。
- **資料庫瀏覽器**（新需求）：新檔 `db_admin.py` 的 `/api/db/browse`、`/api/db/structure`（見 7.7）。**務必做白名單防注入**（表名、排序欄位）；新增此 blueprint 後需在 `__init__.py` 註冊（共用檔，整合時一併處理）。
- **完成標準**：用 email+密碼 `POST /api/admin/login` 可登入；升 admin 會擋掉未設密碼者；公告可推播；手動跑 data_sync 後 `sync-log` 有紀錄；`GET /api/db/structure?table=routes` 回傳欄位結構。

### P4 前端·React 管理後台

- 加 `react-router-dom`；`src/api/client.js`（登入後自動帶 `X-Admin-User`）；`src/auth/AuthContext.jsx`（localStorage + 路由守衛）。
- 頁面：`Login`、`Dashboard`、`Users`（列表 / 停權 / 改 role）、`Announcements`（CRUD + 推播鈕）、`SyncMonitor`（同步狀態表）、`DbBrowser`（資料庫瀏覽：選表 → 分頁/搜尋/排序看資料列 + 結構分頁，呼叫 `/api/db/*`）。
- **完成標準**：`npm start` 可登入後台並操作上述頁面。

### P5 前端·LIFF 使用者頁

- 地圖頁（Google Maps：附近站點 markers + 選定路線 polyline，靜態）。
- 查詢頁（附近站點 / 最快到站 / 收運日程）。
- 收藏管理頁、通知設定頁。
- 資訊頁（垃圾袋規範 + 大型廢棄物 + 公告）。
- **完成標準**：在 LINE 內開 LIFF 各頁，帶 `X-Line-User-Id` 成功呼叫對應 API。

## 7. 前後端接口規格（API Specifications）

> 欄位依實際 schema（ARCHITECTURE.md 第五章）。`*` 為必填。`arrive_time`/`leave_time` 為 `"HH:MM:SS"` 字串；時間戳為 `"YYYY-MM-DD HH:MM:SS"`。布林以 `0/1` 表示。

### 7.1 使用者綁定 `/api/users`

#### 7.1.1 LINE 一鍵綁定
```http
POST /api/users/register
```
Request（`username` 帶入 LINE 暱稱 `profile.displayName`）：
```json
{ "line_user_id": "U1234567890abcdef", "username": "小明" }
```
Response 201：
```json
{ "status": "success", "message": "綁定成功！", "data": { "user_id": 12, "username": "小明" } }
```
規則：`line_user_id` 必填（缺→400）；`username` 帶入 LINE 暱稱（**可重複，不設 UNIQUE**）；此步**不設 email/密碼**（之後由 7.1.3 自助設定）；同一 `line_user_id` 已綁定→409。

#### 7.1.2 取得目前使用者
```http
GET /api/users/me
Header: X-Line-User-Id: U1234567890abcdef
```
Response：
```json
{
  "status": "success",
  "data": { "user_id": 12, "line_user_id": "U1234567890abcdef", "username": "小明", "email": null, "role": "user", "status": "active" }
}
```

#### 7.1.3 設定 / 更新帳號密碼（自助）
> 使用者隨時可設定 email + 密碼；**設定後才有資格被升級為管理者**（見 7.6.3）。
```http
PUT /api/users/credentials
Header: X-Line-User-Id: U1234567890abcdef
```
Request：
```json
{ "email": "me@example.com", "password": "your-password" }
```
Response：`{ "status": "success", "data": null }`
規則：`email`、`password` 必填；`email` **不可重複**（與他人重複→409）；後端以 `generate_password_hash` 將密碼存入 `password_hash`，並更新 `email`。

### 7.6 管理 `/api/admin`

#### 7.6.1 管理員登入
```http
POST /api/admin/login
```
Request：
```json
{ "email": "admin@example.com", "password": "your-password" }
```
Response：

失敗：`{ "status": "error", "message": "帳號或密碼錯誤" }`（401）
規則：以 **email** 查 `users` 且 `role IN ('admin','developer')`、`status='active'`；`check_password_hash(password_hash, password)` 驗證；登入後前端把 **email** 存起來，之後請求帶 `X-Admin-User: <email>`。

#### 7.6.2 使用者列表
```http
GET /api/admin/users?status=active&role=user&q=abc
Header: X-Admin-User: admin@example.com
```
Response：
```json
{
  "status": "success",
  "count": 1,
  "data": [
    { "user_id": 12, "line_user_id": "U123", "username": null, "email": null, "role": "user", "status": "active", "created_at": "2026-05-24 09:00:00" }

```
query 皆選填：`status`、`role`、`q`（對 username/email 模糊查）。

#### 7.6.3 更新使用者（停權 / 改權限）

### 7.7 資料庫瀏覽器 `/api/db`（管理者，需 `X-Admin-User`）

> 後台「看資料庫內容與結構」。**動態 SQL（表名來自前端）有注入風險**，務必遵守下方白名單規則（範例文件未含，本文件補上）。

#### 7.7.1 瀏覽資料列
```http
GET /api/db/browse?table=routes&page=1&limit=50&search=信義&sort=route_id:asc
Header: X-Admin-User: admin@example.com
```
query：`table*`、`page`(預設 1)、`limit`(預設 50，**上限 200**)、`search`(選)、`sort`(選，格式 `欄位:asc|desc`)
Response：
```json
{
  "status": "success",
  "count": 50,
  "total": 13560,
  "page": 1,
  "limit": 50,
  "data": [
    { "route_id": 1, "areas_id": 3, "route_code": "A001", "route_name": "信義1號線", "car_number": "ABC-123", "team": "第一分隊", "trip_number": "1" }
  ]
}
```
（`data` 內欄位＝該表的所有欄位，每張表不同。）

規則（資安，**必做**）：
- `table` 必須在**白名單**（目前 11 張表名）內，否則回 400。
- `sort` 的欄位名必須是該表合法欄位（對照 `DESCRIBE`），方向只允許 `asc`/`desc`。
- `search` 用**參數化** `LIKE %s` 套在文字欄位（`VARCHAR`/`TEXT`），**不可字串拼接**。
- `limit` 轉 int 並夾在 `1..200`（`stations` 有 ~17.5 萬列，禁止無上限撈取）。
- 建議**遮蔽敏感欄位**：`users.password_hash` 不回傳（或以 `"***"` 取代）。

#### 7.7.2 查看資料表結構
```http
GET /api/db/structure?table=routes
Header: X-Admin-User: admin@example.com
```
Response：
```json
{
  "status": "success",
  "data": [
    { "Field": "route_id", "Type": "int(11)", "Null": "NO", "Key": "PRI", "Default": null, "Extra": "auto_increment" },
    { "Field": "areas_id", "Type": "int(11)", "Null": "NO", "Key": "MUL", "Default": null, "Extra": "" },
    { "Field": "route_code", "Type": "varchar(50)", "Null": "YES", "Key": "MUL", "Default": null, "Extra": "" }
  ]
}
```
規則：`table` 同樣需**白名單**；後端對白名單表執行 `DESCRIBE <table>`，回傳元數據（保留 `DESCRIBE` 原欄位 `Field/Type/Null/Key/Default/Extra`，供前端分析主外鍵與型態）。

## 8. 背景任務規格（非 HTTP API）

### 8.1 到站推播 `notifier`
- 觸發：APScheduler 每 60 秒。
- 邏輯：對每筆 `is_active=1` 的 `notifications`，找其站點今日（weekday）的 `station_schedules`；若使用者開啟的類別（`notify_*`）當日有收（`collects_*=1`），且「現在」落在 `[arrive_time − remind_before_mins, arrive_time]` 內，且當日尚未通知過 → `line_service.push_text(line_user_id, 訊息)`。
- 去重：記憶體 `set`，key=`(noti_id, 日期)`（單程序限制，見第 13 節）。

### 8.2 定時同步 `data_sync`
- 觸發：APScheduler 每日 02:00。
- 邏輯：呼叫整併後的 importer 跑 ETL；結束寫一筆 `api_sync_log`（`source` / `status` / `records_affected` / `message` / `started_at` / `finished_at`）。

## 10. 整合測試流程

### 10.1 環境
1. 建 `garbage_database` 並匯入北北基資料。
2. 複製 `example_config.py` → `config.py`，填 DB / LINE 金鑰。
3. 建一個管理者：讓某使用者先設好 email+密碼（7.1.3，或人工在 DB 塞 email + 雜湊密碼），再把其 `role` 改成 `admin`（首位管理者用此人工法）。
4. `python backend/run.py` 啟動；`GET /health` 應回 ok。

### 10.2 使用者流程
1. LINE 輸入「綁定帳號」→ 開 LIFF → 完成綁定 → DB `users` 多一筆（`line_user_id` 有值）。
2. LIFF 查詢頁帶 `X-Line-User-Id` 呼叫 `/api/stations/search` → 看到附近站點與最快到站。
3. 收藏一個站點、設定到站前 5 分鐘提醒。

### 10.3 推播流程
1. 把該站 `arrive_time` 調到「現在 +3 分鐘」、`station_schedules` 當日對應類別設 1。
2. 等 `notifier` 觸發 → 手機收到 LINE 推播。

### 10.4 管理流程
1. `POST /api/admin/login` 成功。
2. 新增公告 → 按推播 → 綁定使用者收到 LINE 群發。
3. 把測試使用者 `status` 設 `suspended` → 該使用者再呼叫使用者 API 應得 403。
4. 手動跑一次 `data_sync` → 後台 SyncMonitor 看到一筆 `api_sync_log`。

## 14. 最後提醒

下一階段的核心不是「每個人都做一點全部」，而是守住邊界：

```text
P1：站點查詢與地理運算
P2：收藏 / 通知 / 資訊查詢
P3：管理後台 API 與背景推播

```

只要保持這個邊界、以第 7 節為唯一接口真相，五人重複改同一檔的機率就會降到最低。
